# Глоссарий Leo4 IoT Platform

Краткий справочник терминов и аббревиатур, используемых в протоколах и документации платформы.

---

## SN (Serial Number)

Серийный номер устройства, используемый как идентификатор в топиках MQTT. Совпадает с **CN** клиентского сертификата mTLS. Используется в шаблоне топиков: `srv/<SN>/...`, `dev/<SN>/...`.

Подробнее: [`docs/mqtt_topic_rules.md`](mqtt_topic_rules.md)

---

## CN (Common Name)

Поле Common Name в X.509‑сертификате клиента. В платформе Leo4 CN = SN устройства. Служит для авторизации устройства при mTLS‑подключении.

Подробнее: [`docs/client-cert-ssl-rmq-config.md`](client-cert-ssl-rmq-config.md)

---

## mTLS (Mutual TLS)

Двусторонняя аутентификация по TLS: и сервер, и клиент предъявляют сертификаты. Используется для аутентификации IoT‑устройств в RabbitMQ. Сертификаты передаются через bind‑mount, не запекаются в Docker‑образ.

Подробнее: [`docs/client-cert-ssl-rmq-config.md`](client-cert-ssl-rmq-config.md)

---

## PKI (Public Key Infrastructure)

Инфраструктура открытых ключей. Используется для управления сертификатами устройств (выпуск, отзыв, проверка CN/SN).

---

## RPC (Remote Procedure Call)

Удалённый вызов процедур. В платформе Leo4 реализован поверх MQTT 5: сервер публикует задачу в топик `srv/<SN>/tsk`, устройство выполняет и публикует ответ в `dev/<SN>/res`. Сопоставление запрос/ответ — через **correlation data**.

Подробнее: [`docs/mqtt-rpc-protocol.md`](mqtt-rpc-protocol.md)

---

## MQTT v5

Пятая версия протокола MQTT с поддержкой correlation data, user properties, response topic и др. В платформе используется через RabbitMQ (плагин MQTT).

Подробнее: [`docs/mqtt-rpc-protocol.md`](mqtt-rpc-protocol.md)

---

## Correlation Data

Бинарное поле в заголовке MQTT 5, которое сервер устанавливает при публикации задачи, а устройство копирует в ответ. Используется для сопоставления RPC‑запроса и ответа.

**Correlation data обязательна** для всех RPC‑взаимодействий.

Подробнее: [`docs/correlation-data-guide.md`](correlation-data-guide.md)

---

## Топики MQTT

| Топик | Направление | Назначение |
|---|---|---|
| `srv/<SN>/tsk` | Сервер → Устройство | Задача (task) |
| `srv/<SN>/rsp` | Сервер → Устройство | Ответ сервера |
| `srv/<SN>/cmt` | Сервер → Устройство | Команда (command) |
| `srv/<SN>/eva` | Сервер → Устройство | Оценка (evaluation) |
| `dev/<SN>/res` | Устройство → Сервер | Результат задачи |
| `dev/<SN>/evt` | Устройство → Сервер | Событие (event) |

`<SN>` = CN сертификата устройства.

Подробнее: [`docs/mqtt_topic_rules.md`](mqtt_topic_rules.md)

---

## method_code

Числовой код метода (команды), которую нужно выполнить на устройстве. Передаётся в payload задачи. Реестр кодов: [`docs/method-codes-reference.md`](method-codes-reference.md).

---

## event_type_code

Числовой код типа события, публикуемого устройством. Например, `13`/`14` — подтверждение физического выполнения задачи. Реестр: [`docs/event-types-reference.md`](event-types-reference.md).

---

## Tag 304 (cell number)

Числовой тег в payload события, означающий номер ячейки (cell number). Используется для идентификации конкретного физического слота/ячейки устройства.

Подробнее: [`docs/event-property-tags.md`](event-property-tags.md)

---

## Статусы задачи

| Статус | Код | Описание |
|---|---|---|
| `READY` | 0 | Задача создана, ожидает отправки |
| `PENDING` | 1 | Задача отправлена устройству |
| `LOCK` | 2 | Устройство приступило к выполнению |
| `DONE` | 3 | Сервер получил ответ от устройства |
| `EXPIRED` | 4 | TTL истёк (TTL=0) |
| `DELETED` | 5 | Задача удалена |
| `FAILED` | 6 | Ошибка выполнения |

> **Важно:** `DONE` (status=3) означает, что сервер получил ответ, но **не** подтверждает физическое выполнение. Физическое подтверждение — только через события с `event_type_code` 13/14.

Подробнее: [`docs/task_states.md`](task_states.md)

---

## TTL (Time To Live)

Счётчик «живучести» задачи. Декрементируется по правилам платформы. При TTL=0 задача переходит в статус `EXPIRED`.

Подробнее: [`docs/TTL.md`](TTL.md)

---

## Webhook

Механизм push‑уведомлений: сервер отправляет HTTP POST‑запрос на заранее настроенный URL при изменении статуса задачи или получении события.

Подробнее: [`docs/3-webhooks.md`](3-webhooks.md)
