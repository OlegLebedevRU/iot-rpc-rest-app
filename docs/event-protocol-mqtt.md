# Протокол MQTT-сообщений для отправки событий (evt)

> **Файл:** `docs/event-protocol-mqtt.md`  
> **Версия:** 0.50  
> **Дата:** 2026  
> **Тема:** Формат асинхронных событий в топике `dev/<SN>/evt`

---

## Назначение

Данный документ описывает формат **MQTT-сообщений для отправки асинхронных событий** от устройства (приложения или контроллера) в облако через топик:

где `<SN>` — серийный номер устройства (уникальный идентификатор).

Пример sn: **a3b1234567c10221d290825**

События (`evt`) используются для передачи информации о внутренних изменениях состояния устройства: срабатывание датчиков, действия пользователя, системные уведомления и пр.

---

## Общая структура сообщения

Сообщение представляет собой **JSON-объект**, содержащий поля с числовыми тегами. Поддерживается единый формат для всех типов событий.

### Основные поля:

| Тег | Ключ | Тип | Описание |
|-----|------|-----|--------|
| `101` | `"101"` | `uint32` | Уникальный идентификатор события на стороне устройства (в пределах сессии или перезагрузки) |
| `102` | `"102"` | `string` | Временная метка создания события в формате ISO 8601. Может содержать часовой пояс (например, `+03:00`, `Z`, `ZMSK`). |
| `200` | `"200"` | `int` | Код типа события (см. таблицу ниже) |
| `300` | `"300"` | `array` | Массив анонимных JSON-объектов с параметрами события. Структура зависит от значения `200`. |

> Поле `"300"` всегда является массивом, даже если содержит один объект.  
> Каждый объект внутри `"300"` состоит из именованных элементов, где имя элемента — числовой тег (`3xx`, `4xx` и т.п.), определённый в [справочнике тегов свойств событий](./event-property-tags.md).  
> Значение элемента может иметь любой допустимый JSON-тип и определяется спецификацией соответствующего тега.

## Примеры сообщений

### Пример 1: Событие "Датчик замка (открытие)" (`200: 13`)
```json 
Вариант 1, краткий вариант без дополнительных полей:
{
    "101": 20338,
    "300": [{ "304": 12 }],
    "102": "2026-02-18T01:43:16ZMSK",
    "200": 13
}        

Вариант 2, с данными контроллера:
{
    "101": 20338,
    "300": [{ "304": 1, "306": 22, "305": 1, "307": 35921924, "308": 8392849 }],
    "102": "2026-02-18T01:43:16ZMSK",
    "200": 13
}
```
> **Интерпретация примера 1:** 
- Вариант 1 - ПО устройства определило событие «Открыто» датачика #12 (`304: 12`). Источник — hardware (замковая плата).
- Вариант 2 - ПО устройства определило событие «Открыто» датачика #1 (`304: 1`). Дополнительно указаны параметры адреса контроллера (`305: 1`) и ID канала (`306: 22`), а также значения битового регистра контроллера каналов ДО (`307: 35921924`) и ПОСЛЕ (`308: 8392849`) события.

### Пример 2: Событие "Ввод идентификатора (карта, пин и т.п.)" (`200: 3`)
````json 
{ "101": 36823, "102": "2026-03-03T12:41:33ZMSK", "200": 3, "300": [ { "301": "044AFE42C76781", "302": 6, "303": 0 } ] }

````
> **Интерпретация:** Пользователь ввёл идентификатор (например, карту), который не найден в локальной базе (`303: 0`). Источник — устройство ввода типа 6 (NFC считыватель).

### Пример 3: Событие "keep-alive" (`200: 44`):

## Передача по MQTT

### Публикация события

Устройство публикует сообщение в топик:
____
`/dev/{sn}/evt`
____


#### Пример:
```json
- Topic: dev/a3b1234567c10221d290825/evt 
- QoS: 1 
- Retain: 0
- Payload:         
{ "101": 36823, "102": "2026-03-03T12:41:33+03:00", "200": 3, "300": [ { "301": "044AFE42C76781", "302": 6, "303": 0 } ] }
- User Properties:
  -- event_type_code: 3
  -- dev_event_id: 36823
  -- dev_timestamp: 1740984093 
  -- correlation_id: d9afcbfa-3d2c-4304-8e69-f644fef29f1f
```

> **Форматы `dev_timestamp` в User Properties:**
> Сервер принимает значение `dev_timestamp` в нескольких форматах:
> - **Unix epoch** (целое число или строка): `1740984093`
> - **ISO 8601** (строка с часовым поясом): `2026-03-03T12:41:33+03:00`
> - Если значение отсутствует или не может быть разобрано — используется текущее серверное время.

> **Рекомендации:**
> - Использовать **QoS 1** для гарантированной доставки.
> - Заполнять `User Properties` для фильтрации на сервере.
> - Генерировать уникальный `correlation_id` (UUID4) для каждого события.

---

### Подтверждение получения (EVA)

После обработки события сервер отправляет подтверждение в топик `srv/<SN>/eva`.

#### Условия отправки EVA

EVA отправляется при выполнении **всех** условий:
- `event_type_code` задан и не равен `0`
- `event_type_code` не является gauge-типом (фоновой телеметрией)
- `dev_event_id` задан и не равен `0`

#### Идемпотентность

Сервер обеспечивает **идемпотентную обработку** событий. Дубликаты определяются по составному уникальному ключу `(device_id, dev_event_id, dev_timestamp)` (при условии `dev_event_id != 0`):

- **Новое событие** — сохраняется в БД, публикуется вебхук.
- **Дубликат** — не сохраняется повторно, вебхук не публикуется.
- **EVA отправляется в обоих случаях** — и для новых, и для дублированных событий, обеспечивая устройству подтверждение получения.

#### Формат EVA-сообщения

```text
Topic: srv/a3b1234567c10221d290825/eva
QoS: 0
CorrelationData: <UUID из исходного evt>
User Properties:
  -- event_type_code: 13
  -- dev_event_id: 20338
  -- correlationData: d9afcbfa-3d2c-4304-8e69-f644fef29f1f
Payload: {"status": "success"}
```

| Payload `status` | Описание |
|-----------------|----------|
| `"success"` | Событие успешно обработано (или идемпотентный дубликат) |
| `"error"` | Произошла ошибка при сохранении события |

> **Примечание:** EVA-сообщения имеют TTL = 180 секунд (3 минуты) на уровне брокера.

## Рекомендации по реализации

- Все временные метки (`102`) в payload должны быть в UTC или с явным указанием зоны (`Z`, `+03:00`, `ZMSK` и т.п.).
- Значение `dev_timestamp` в User Properties может быть **Unix epoch** (целое число) или **ISO 8601** строкой — сервер корректно обрабатывает оба формата.
- Идентификатор `101` должен быть уникален в пределах активной сессии устройства.
- Пара `(dev_event_id, dev_timestamp)` используется для идемпотентной дедупликации событий на сервере.
- Сообщения должны быть валидными JSON без лишних пробелов (для экономии трафика).
- При отправке через медленные каналы — использовать сжатие или бинарную сериализацию (в будущем).

---

## Заключение

Протокол передачи событий через `dev/<SN>/evt` обеспечивает:
- Единый формат данных
- Гибкость за счёт кодирования типов
- Поддержку множества сценариев (от датчиков до системного мониторинга)
- Совместимость с MQTT 5 и User Properties
- Идемпотентную обработку событий на стороне сервера
- Подтверждение получения (EVA) через топик `srv/<SN>/eva`

Данный подход позволяет централизованно обрабатывать события от тысяч устройств в облаке.
