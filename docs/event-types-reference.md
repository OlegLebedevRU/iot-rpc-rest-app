# Справочник MQTT-событий (evt) и типов (тег 200)

> **Файл:** `docs/event-types-reference.md`
> **Назначение:** Референсный документ (источник истины) по типам событий (поле `200`), отправляемым устройством в облако.
> **Важно:** В этом файле не приводится полная расшифровка числовых тегов внутри массива `300` (теги `3xx`, `4xx`). Их назначения вынесены в единый референс-документ: [`event-property-tags.md`](./event-property-tags.md).

---

## 1. Общий формат события

Все события от устройства (в топик `dev/<SN>/evt`) публикуются в виде JSON-объекта со следующей базовой структурой:

```json
{
  "101": 12345,
  "102": "2026-04-22T13:48:02+03:00",
  "200": 0,
  "300": [
    { ... }
  ]
}
```

- **`101`** (uint32): Уникальный идентификатор события на ��стройстве (в пределах сессии/аптайма).
- **`102`** (string): Временная метка ISO 8601 (с часовым поясом).
- **`200`** (int): Код типа события (см. таблицу ниже). Полностью определяет структуру вложенных параметров.
- **`300`** (array): Массив объектов с параметрами события. Структура содержимого зависит от типа `200`.

## 2. Справочник-таблица типов событий (Код 200)

> **Матрица совместимости** фиксирует, для каких вариантов устройств данный тип события задокументирован: `Platerra`, `Siplite`, `l4-hmi`.

| :--- | :--- | :--- | :--- |

| **2** | `DoorSensorState` | Изменение состояния GPIO-входов (например, массив замков/дверей открыт или закрыт). | `Platerra`, `Siplite` |
| **3** | `IdInput` | Пользователь ввел идентификатор (RFID-карта, PIN-код, биометрия). | `Siplite` |
| **13** | `CellOpenEvent` | Событие GPIO/Входа: открытие конкретной ячейки. | `Platerra`, `Siplite` |
| **14** | `CellCloseEvent` | Событие GPIO/Входа: закрытие конкретной ячейки. | `Platerra`, `Siplite` |
| **23** | `CellBindedAct` | Привязка логического номера ячейки к аппаратному каналу (hardware контроллеру). | `Siplite` |

| **70** | `L4HmiEvent` | **(НОВОЕ)** Результат задания по обновлению UI-каталога устройств варианта совместимости `l4-hmi`. | `l4-hmi` |

## 3. Полные форматы известных событий

### Код 0: `Hello`
Первичная инициализация устройства.
```json
{
  "101": 1,
  "102": "2026-04-22T10:00:00+00:00",
  "200": 0,
  "300": [
    {
      "310": "1.05.000",
      "311": 15,
      "314": "aa:bb:cc:dd:ee:ff",
      "323": "192.168.1.50",
      "324": "sn12345678",
      "340": { "state": "SIP_IDLE", "diag": { "cnt": {}, "cf": 0, "crit": [] } }
    }
  ]
}
```

### Код 3: `IdInput`
Срабатывание устройства ввода (например, RFID).
```json
{
  "101": 36823,
  "102": "2026-03-03T12:41:33+03:00",
  "200": 3,
  "300": [
    {
      "301": "044AFE42C76781",
      "302": 6,
      "303": 0
    }
  ]
}
```

### Коды 13 / 14: `CellOpenEvent` / `CellCloseEvent`
Событие открытия (13) или закрытия (14) ячейки.
```json
{
  "101": 20338,
  "102": "2026-02-18T01:43:16+03:00",
  "200": 13,
  "300": [
    {
      "304": 12,
      "305": 1,
      "306": 22,
      "307": 35921924,
      "308": 8392849
    }
  ]
}
```

### Код 27: `OtaResult`
Результат загрузки и применения OTA-обновления.
```json
{
  "101": 1500,
  "102": "2026-04-20T11:00:00+03:00",
  "200": 27,
  "300": [
    {
      "341": "OTA_OK",
      "342": "Update successful"
    }
  ]
}
```

### Код 44: `DevHelthCheck`
Регулярный пинг/статус устройства. Формат аналогичен `Hello` (Код 0).
```json
{
  "101": 1047,
  "102": "2026-08-12T10:31:57+00:00",
  "200": 44,
  "300": [
    {
      "310": "1.04.025",
      "313": 65740,
      "324": "a3b1234567c10221d290825",
      "339": "3600",
      "340": { "state": "SIP_REGISTERED", "diag": { "cnt": {}, "cf": 0, "crit": [] } }
    }
  ]
}
```

### Код 54: `SipRegStateChange`
Изменение состояния SIP-субсистемы устройства.
```json
{
  "101": 20456,
  "102": "2026-04-02T14:23:45+03:00",
  "200": 54,
  "300": [
    {
      "353": "LOST_REGISTERED",
      "354": "ip_lost",
      "340": { "state": "SIP_UNREGISTERED", "diag": { "cnt": {}, "cf": 1, "crit": [] } }
    }
  ]
}
```

### Код 63: `FullStateNotify` (Текущее бинарное состояние всех ячеек)
Новое событие для передачи полного слепка состояния (открыто/закрыто, активно/неактивно) всех подконтрольных ячеек, слотов и GPIO.
```json
{
  "101": 502,
  "102": "2026-04-22T13:48:02+03:00",
  "200": 63,
  "300": [
    {
      "360": [
        { "304": 1, "321": 1 },
        { "304": 2, "321": 1 },
        { "304": 3, "321": 0 }
      ]
    }
  ]
}
```

### Код 65: `SeedInitDone` (Завершение seed-инициализации NVS)

Событие отправляется устройством (siplite / ESP32) **однократно** после успешного завершения seed-flow при первом запуске или после factory-reset. Несёт слепок всех дефолтных значений, записанных в NVS-партиции (`cfg`, `db1`), в теге `370` (`seed_data`).

> **Важно:** событие формируется и отправляется исключительно siplite (ESP32); контракта с STM32 у этого события нет.

#### Формат

```json
{
  "101": 1,
  "102": "2026-04-26T09:00:00+00:00",
  "200": 65,
  "300": [
    {
      "310": "1.05.001",
      "314": "aa:bb:cc:dd:ee:ff",
      "324": "sn12345678",
      "370": [
        { "p": "cfg", "ns": "cfg_eth", "k": "eth_port_mode", "t": "i32", "v": 0 },
        { "p": "cfg", "ns": "cfg_net",  "k": "dhcp_enabled",  "t": "i32", "v": 1 },
        { "p": "db1", "ns": "app_cfg",  "k": "log_level",     "t": "i32", "v": 2 }
      ]
    }
  ]
}
```

#### Поле `370` (`seed_data`)

Тег `370` содержит **массив объектов NVS-записей**, каждый из которых идентичен по структуре элементам массива `dt` команды `CMD_SET_NVS_RECORD` (`method_code 49`) и строкам ответа `CMD_GET_NVS_RECORD` (`method_code 50`).

| Поле | Тип | Описание |
|------|-----|----------|
| `p` | `string` | Название NVS-партиции (`"cfg"`, `"db1"` и т.п.) |
| `ns` | `string` | Namespace внутри партиции |
| `k` | `string` | Ключ NVS-записи |
| `t` | `string` | Тип значения: `"i32"`, `"str"`, `"blob"` и т.п. |
| `v` | any | Значение записи; тип определяется полем `t` |

#### Совместимость с NVS RPC (`method_code 49 / 50`)

Структура каждого объекта внутри `370` (`seed_data`) **полностью совпадает** с форматом, используемым в:

- `method_code 49` (`CMD_SET_NVS_RECORD`) — запись значений в NVS (поля `p`, `ns`, `k`, `t`, `v`)
- `method_code 50` (`CMD_GET_NVS_RECORD`) — чтение записей из NVS (ответ возвращает те же поля)

Это позволяет серверной стороне:

1. При получении `SeedInitDone` сохранить слепок как baseline конфигурации устройства.
2. Сравнивать `seed_data` с результатами последующих вызовов `CMD_GET_NVS_RECORD` для детектирования дрейфа конфигурации.
3. При необходимости восстановить дефолты, передав записи из `seed_data` обратно через `CMD_SET_NVS_RECORD`.

### Код 70: `L4HmiEvent`

Событие публикуется устройством `l4-hmi` как **асинхронный результат задания по обновлению UI-каталога**.

#### Краткая карточка

| Поле | Значение |
|------|----------|
| `event_type_code` | `70` |
| Назначение | Результат попытки скачать, провалидировать и/или активировать новый UI-каталог |
| Совместимость | `l4-hmi` |
| Основные теги | `401`–`410` |

#### Формат

```json
{
  "101": 781,
  "102": "2026-04-29T12:30:35+00:00",
  "200": 70,
  "300": [
    {
      "401": "parse_error",
      "402": 3,
      "403": 2,
      "404": 2,
      "405": "a",
      "406": "1fe45a8304b6e61d1518d71d8dae57520a4d11a81046b179a434474197bb29be",
      "407": 110208,
      "408": 750,
      "409": "074d2f69-289f-4a3f-904b-b745e2f75364",
      "410": "nodes[8] name too long"
    }
  ]
}
```

#### Расшифровка тегов

| Тег | Тип | Описание |
|-----|-----|----------|
| `401` | `string` | Строковый код результата / ошибки обновления. Пример: `"parse_error"` — файл скачан, но JSON/валидация каталога отвергнута парсером. |
| `402` | `int` | Версия UI-каталога, которую устройство пыталось применить. |
| `403` | `int` | Активная версия UI-каталога **до** попытки обновления. |
| `404` | `int` | Активная версия UI-каталога **после** попытки обновления. |
| `405` | `string` | Активный слот UI-каталога после завершения операции (например, `"a"`). |
| `406` | `string` | SHA-256 скачанного файла UI-каталога (hex-строка). |
| `407` | `uint32` | Размер скачанного файла UI-каталога в байтах. |
| `408` | `uint32` | Длительность операции обновления в миллисекундах. |
| `409` | `string` | `correlation_id` исходного RPC-задания, по которому было инициировано обновление. |
| `410` | `string` | Диагностическая причина или текст ошибки для журналирования / поддержки. |

#### Интерпретация примера

В примере устройство:

1. Получило задание применить каталог версии `3`.
2. Скачало файл и вычислило его SHA-256 / размер.
3. Не смогло принять каталог из-за ошибки парсинга (`401 = "parse_error"`).
4. Сохранило прежнюю активную версию `2` и слот `"a"`.
5. Передало в `409` ссылку на исходный RPC через `correlation_id`, чтобы backend мог сопоставить событие с задачей.
