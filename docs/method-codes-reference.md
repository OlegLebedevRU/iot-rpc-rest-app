# Справочник MQTT RPC Method Codes

Данный документ представляет собой единый реестр всех `method_code`, используемых в приложении, объединяющий константы из исходного кода (`components/l4_types/include/l4_base_types.h`) и документации по потокам данных.

## 1. Концепция и Диапазоны

Значение `method_code` определяет действие (команду), которое должно выполнить устройство в рамках RPC-запроса от сервера.

- **Стандартные коды (1 - 2999):** Обрабатываются во внутренних подсистемах прошивки ESP32 (OTA, NVS, UART, загрузка/сброс данных).
- **Интерактивные коды (3000 - 3999):** Предназначены для сквозной передачи запросов на внешние шлюзы (через WebSocket). Для интерактивных `method_code` устройство **обязано** завершать вызов в терминальное состояние (сразу публиковать `/res` со статусом `500`) даже при недоступности внешнего WS-клиента (fail-fast поведение).
- **Служебные терминальные коды (например, 0xFFFF):** Обрабатываются как fail-fast.

## 3. Детализация форматов запросов и ответов (Payload dt)

Ниже описаны структуры массивов `dt` (Data) для поля `payload.dt` для каждой команды. Если ответ (`res`) не содержит специальных полей, генерируется стандартный статус `{"status": "OK", "method_code": <code>}`. Для всех сообщений обязательна сквозная передача `correlationData`.

### 15 `CMD_LOAD_CELLS_FROM_CLOUD`
**Запрос (`dt`):** Формат зависит от логики парсинга ячеек (массив объектов свойств ячеек).

### 16 `CMD_LOAD_CARDS_FROM_CLOUD`
**Запрос (`dt`):** Ожидаются объекты привязок `[{"cd": "номер_карты", "cl": номер_ячейки}]`.

### 18 `CMD_SEND_485`
**Запрос (`dt`):** `[{"uartNum": 2, "txBase64": "base64_string"}]`
Отправляет бинарные данные в указанный UART/RS-485.

### 19 `CMD_SW_UPDATE` (Обновление прошивки ESP32)
**Запрос (`dt`):** `[{"url": "https://..."}]`
Начинает скачивание бинарника ESP32 и обновление (OTA).
*Ошибки (res)*: 400 (невалидный URL/пустой объект), 409 (OTA уже идет), 500 (внутренняя ошибка).

### 20 `CMD_SUB_REQUEST` (��акетные команды / Sub-events)
**Запрос (`dt`):** `[{"mt": EventTypeNumber}, ...]`
Генерирует исходящие события (`evt` со статусом 206) во внутреннюю шину. Поддерживает события: `0` (HELLO_FROM_DEV), `4` (COPY_DATA_CELLS_TO_CLOUD), `5`, `25` (SEARCH_CARDS_BY_CELL с полем `"cl"`), `28`, `37`.

### 21 `CMD_ESP_RESTART`
**Запрос (`dt`):** Может быть пустым (игнорируется).
Производит мгновенный вызов `esp_restart()`.

### 35 `CMD_API_CODE_INPUT` (Ввод кода)
**Запрос (`dt`):** `["12345", "67890"]` (массив ASCII-строк, длина не более 15 символов).
Генерирует программные события чтения карт/кодов.
*Ответ (res)*: Для каждой строки генерируется индивидуальный ответ через функцию вывода результатов с `status_code`: 200 (`accepted`), 423 (`dropped_invalid` ил `dropped_busy`).

### 42 `CMD_ASCII_TO_UART`
**Запрос (`dt`):** Публикует переданные в `"tx"` строки напрямую на внешний интерфейс (ожидается расширение стандарта `"tx": "строка"`).

### 48 `CMD_SEND_RAW`
**Запрос (`dt`):** `[{"type": "udp", "port": 62778, "txBase64": "base64_string"}]`

### 49 `CMD_SET_NVS_RECORD`
**Запрос (`dt`):**
```json
[{ "p": "cfg", "ns": "cfg_eth", "k": "eth_port_mode", "t": "i32", "v": 0 }]
```
Детали и типы (`str`, `i32` и т.д.) см. в `mqtt-rpc-protocol.md`.
*Ответ (res)*: `{"status": "OK", "method_code": 49, "records_updated": 1}`. 

### 50 `CMD_GET_NVS_RECORD`
**Запрос (`dt`):** `[{"ns": "cfg_eth"}]`
*Ответ (res)*: Возвращает сырой массив записей: `[{"k": "...", "t": "...", "v": ...}]`.

### 51 `CMD_OPERATE_CELL` (Управление ячейкой / Cell Operation)
Запрос от сервера включает параметр `cl` (номер ячейки) в массиве `dt`.

**Запрос (dt):**
```json
[
  {
    "cl": 1
  }
]
```

**Ожидаемый ответ (Response payload):**
В ответе `/res` обязательно должны присутствовать поля, подтверждающие обработку конкретной ячейки:
```json
{
  "cl": 1,
  "ts": "2026-04-20T13:17:30.798282+00:00",
  "result": "ok",
  "accepted": true,
  "method_code": 51
}
```

### 411 `CMD_STM32_MCU_UPDATE`
Запускает обновление прошивки сопроцессора STM32 по шине из локальной базы.

### 512 `CMD_STM32_BIN_DOWNLOAD`
**Запрос (`dt`):** 
```json
[
  {
    "url": "https://...",
    "sha256": "64_hex_chars_hash",
    "chunk_size": 1024
  }
]
```
*Ответ (res)*: При успешном старте `{"status": "OK","method_code": 512, "detail": "download task started"}`. При ошибках возвращаются коды 400 (плохие аргументы), 409 (уже идет).

### 3000..3999 (Interactive WS Gateway)
Сквозные вызовы, payload и структура определяются подключенным клиентом. Отличие - fail-fast при недоступности WS-канала.
