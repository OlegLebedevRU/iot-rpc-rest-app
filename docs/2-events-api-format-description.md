## 🎯 Назначение и контекст

Данный документ — **главный справочник разработчика REST API** для работы с событиями устройств IoT-платформы. Он описывает формат данных, доступные эндпоинты и паттерны интеграции.

### Путь события: от устройства до API

```
Устройство  →  MQTT топик dev/<SN>/evt  →  Сервер  →  База данных  →  REST API
```

Сервер получает события из MQTT-топика, сохраняет их **в оригинальном виде** (без переименования полей и без преобразования структуры) и предоставляет через REST API. Поле `payload` в ответе — точная копия MQTT-сообщения от устройства.

### REST API vs MQTT: в чём разница?

| Аспект | MQTT | REST API |
|--------|------|----------|
| Режим получения | Push (подписка на топик) | Pull (периодический запрос) |
| Данные | Потоковые, в реальном времени | Хранимые, исторические |
| Фильтрация | На уровне топика и User Properties | По `device_id`, `event_type_code`, диапазону дат |
| Документ | [`event-protocol-mqtt.md`](./event-protocol-mqtt.md) — для разработчиков firmware | **Этот документ** — для разработчиков API-клиентов |

> 💡 Если вам нужны события в реальном времени без polling — рассмотрите [вебхуки](./3-webhooks.md).
> Если вы разрабатываете firmware устройства — ваш основной документ: [`event-protocol-mqtt.md`](./event-protocol-mqtt.md).

## 🗂️ Структура события (DevEventOut)

Каждое событие, возвращаемое API — это JSON-объект с **метаданными сервера** и **оригинальным payload от устройства**.

### Поля объекта DevEventOut

| Поле | Тип | Источник | Описание |
|------|-----|----------|----------|
| `id` | `int` | Сервер | Уникальный ID события (автоинкремент) — используется как курсор в `incremental` |
| `device_id` | `int` | Сервер | Внутренний ID устройства в системе |
| `sn` | `string` | MQTT топик | Серийный номер устройства (`<SN>`) |
| `event_type_code` | `int` | MQTT User Properties / тег `200` | Код типа события — главный ключ для фильтрации |
| `timestamp` | `datetime` | MQTT User Properties / тег `102` | Время создания события на устройстве (ISO 8601 с TZ) |
| `received_at` | `datetime` | Сервер | Время получения события сервером (UTC) |
| `dev_event_id` | `int` | MQTT User Properties / тег `101` | ID события на устройстве (используется для идемпотентности) |
| `correlation_id` | `string` | MQTT User Properties | UUID события |
| `payload` | `object` | MQTT Payload | **Оригинальное JSON-сообщение от устройства — без изменений** |

### Полный пример DevEventOut

```json
{
  "id": 150283,
  "device_id": 4712,
  "sn": "a3b1234567c10221d290825",
  "event_type_code": 13,
  "timestamp": "2026-02-18T01:43:16+03:00",
  "received_at": "2026-02-18T01:43:17Z",
  "dev_event_id": 20338,
  "correlation_id": "d9afcbfa-3d2c-4304-8e69-f644fef29f1f",
  "payload": {
    "101": 20338,
    "102": "2026-02-18T01:43:16ZMSK",
    "200": 13,
    "300": [
      {
        "304": 12,
        "305": 1,
        "306": 22,
        "307": 35921924,
        "308": 8392849
      }
    ]
  }
}
```

> ⚠️ `payload` — точная копия MQTT-сообщения, без изменений. Поле `timestamp` на уровне API нормализовано к ISO 8601 и может отличаться от строки внутри `payload["102"]`.

## 📦 Формат payload

Поле `payload` содержит оригинальное MQTT-сообщение от устройства с **числовыми тегами**.

### Корневые теги payload

| Тег | Тип | Соответствие в DevEventOut | Описание |
|-----|-----|---------------------------|----------|
| `"101"` | `uint32` | `dev_event_id` | ID события на устройстве |
| `"102"` | `string` | `timestamp` (нормализован) | Временная метка ISO 8601 (может содержать `+03:00`, `Z`, `ZMSK`) |
| `"200"` | `int` | `event_type_code` | Код типа события |
| `"300"` | `array` | — | Массив объектов с параметрами события |

### Параметры события — тег `"300"`

Массив `"300"` содержит **анонимные JSON-объекты** с числовыми тегами вида `3xx`. Набор тегов зависит от типа события (`"200"`):

```
payload["300"][0]["304"]  →  номер ячейки (логический)
payload["300"][0]["321"]  →  состояние: 0 = закрыто, 1 = открыто
payload["300"][0]["301"]  →  идентификатор (код карты, PIN)
```

> 📖 **Полный справочник тегов `3xx`** — в [`event-property-tags.md`](./event-property-tags.md) (**единственный источник истины по тегам**). Там же описаны диапазоны тегов, их типы и совместимость по типам устройств.
>
> 📖 **Полные форматы payload для каждого типа события** — в [`event-types-reference.md`](./event-types-reference.md).

## 🔌 API-эндпоинты

**Базовый URL:** `https://<host>/api/v1`
**Аутентификация:** заголовок `x-api-key: ApiKey <key>`

---

### 1. `GET /api/v1/device-events/` — список событий с пагинацией

Получение страницы событий по устройству с возможностью фильтрации по типу.

#### Параметры запроса

| Параметр | Тип | Обязательный | Описание |
|----------|-----|:---:|----------|
| `device_id` | `int` | ✅ | ID устройства |
| `events_include` | `list[int]` | — | Включить только указанные `event_type_code` |
| `events_exclude` | `list[int]` | — | Исключить указанные `event_type_code` |

> 💡 Коды `event_type_code` для `events_include` / `events_exclude` берутся из [таблицы типов событий](#-типы-событий-event_type_code).

#### Пример запроса

```bash
# Получить только события открытия и закрытия ячеек для устройства
curl -X GET \
  'https://dev.leo4.ru/api/v1/device-events/?device_id=4712&events_include=13&events_include=14' \
  -H 'x-api-key: ApiKey <key>'
```

#### Формат ответа

```json
{
  "items": [
    {
      "id": 150283,
      "device_id": 4712,
      "sn": "a3b1234567c10221d290825",
      "event_type_code": 13,
      "timestamp": "2026-02-18T01:43:16+03:00",
      "received_at": "2026-02-18T01:43:17Z",
      "dev_event_id": 20338,
      "correlation_id": "d9afcbfa-3d2c-4304-8e69-f644fef29f1f",
      "payload": { "101": 20338, "102": "...", "200": 13, "300": [{ "304": 12 }] }
    }
  ],
  "total": 150,
  "page": 1,
  "size": 50,
  "pages": 3
}
```

### 2. `GET /api/v1/device-events/incremental` — инкрементальные события

Получение новых событий с момента последнего запроса. **Основной режим для polling-интеграций.**

#### Параметры запроса

| Параметр | Тип | Обязательный | Описание |
|----------|-----|:---:|----------|
| `device_id` | `int` | — | ID устройства. Если не указан — события по всем устройствам организации |
| `last_event_id` | `int` | — | Последний известный `id` события. Если не указан — используется сохранённый серверный оффсет |
| `limit` | `int` (1–100) | — | Максимум событий в ответе (по умолчанию 50) |

#### Пример запроса

```bash
# Получить новые события после ID 150283, не более 100 штук
curl -X GET \
  'https://dev.leo4.ru/api/v1/device-events/incremental?device_id=4712&last_event_id=150283&limit=100' \
  -H 'x-api-key: ApiKey <key>'
```

#### Формат ответа

```json
[
  {
    "id": 150284,
    "device_id": 4712,
    "sn": "a3b1234567c10221d290825",
    "event_type_code": 14,
    "timestamp": "2026-02-18T01:45:00+03:00",
    "received_at": "2026-02-18T01:45:01Z",
    "dev_event_id": 20339,
    "correlation_id": "a1b2c3d4-...",
    "payload": { "101": 20339, "102": "...", "200": 14, "300": [{ "304": 12 }] }
  }
]
```

> 🔄 **Паттерн polling:** Сохраняйте `last_event_id` локально. При следующем запросе передавайте его — получите только новые события. События приходят в порядке возрастания `id`.
>
> 💾 **Восстановление после сбоя:** Если `last_event_id` не передан, сервер использует сохранённый серверный оффсет. Рекомендуется всегда сохранять `last_event_id` на стороне клиента.

### 3. `GET /api/v1/device-events/fields/` — выборка значений тега

Извлечение значений конкретного **числового тега** из `payload` за заданный интервал времени. Оптимален для построения временных рядов и точечного polling конкретного типа события.

#### Параметры запроса

| Параметр | Тип | Обязательный | Описание |
|----------|-----|:---:|----------|
| `device_id` | `int` | ✅ | ID устройства |
| `event_type_code` | `int` | ✅ | Тип события (`200`) для фильтрации |
| `tag` | `str` | ✅ | Числовой тег для извлечения из `300[0]` (например, `"304"`) |
| `interval_m` | `int` | ✅ | Глубина выборки в минутах (назад от текущего момента) |
| `limit` | `int` | — | Лимит записей (макс. 1000) |

#### Пример: polling открытий ячеек за последние 5 минут

```bash
# event_type_code=13 (CellOpenEvent), tag=304 (номер ячейки)
curl -X GET \
  'https://dev.leo4.ru/api/v1/device-events/fields/?device_id=4617&event_type_code=13&tag=304&interval_m=5&limit=10' \
  -H 'accept: application/json' \
  -H 'x-api-key: ApiKey <key>'
```

#### Формат ответа

```json
[
  {
    "created_at": "2026-02-18T01:43:16+03:00",
    "value": 12,
    "interval_sec": 8
  },
  {
    "created_at": "2026-02-18T02:15:08+03:00",
    "value": 5,
    "interval_sec": 41
  }
]
```

- `value` — значение тега `304` из первого объекта в `300[]`
- `interval_sec` — интервал с предыдущим событием того же типа

> 💡 **Когда использовать `fields/` вместо `incremental`:**
> - Нужны только значения одного конкретного тега (например, `304` — номер открытой ячейки)
> - Нужна история за период без курсора
> - Строите временной ряд для аналитики или графиков
>
> Для приёма потока всех событий по устройству используйте [`incremental`](#2-get-apiv1device-eventsincremental--инкрементальные-события).

## 📋 Типы событий (event_type_code)

Значение `event_type_code` в ответе API соответствует тегу `"200"` внутри `payload`. Это **главный ключ для фильтрации и маршрутизации событий** на стороне клиента.

| Код | Тип | Описание | Основные теги в `300` | Совместимость |
|:---:|-----|----------|-----------------------|---------------|
| `0` | `Hello` | Первичная инициализация при подключении | `310`, `314`, `323`, `324`, `340` | `Platerra`, `Siplite` |
| `2` | `DoorSensorState` | Изменение состояния GPIO-входов | `304`, `321` | `Platerra`, `Siplite` |
| `3` | `IdInput` | Ввод идентификатора (RFID, PIN, биометрия) | `301`, `302`, `303` | `Siplite` |
| `13` | `CellOpenEvent` | Открытие ячейки | `304`, `305`, `306`, `307`, `308` | `Platerra`, `Siplite` |
| `14` | `CellCloseEvent` | Закрытие ячейки | `304`, `305`, `306`, `307`, `308` | `Platerra`, `Siplite` |
| `23` | `CellBindedAct` | Привязка ячейки к аппаратному каналу | `304`, `305`, `306` | `Siplite` |
| `27` | `OtaResult` | Результат OTA-обновления прошивки ESP32 | `341`, `342`, `343`–`346` | `Siplite` |
| `38` | `BoardConnected` | Плата расширения подключена | — | `Siplite` |
| `39` | `BoardDisconnected` | Плата расширения отключена | — | `Siplite` |
| `44` | `DevHelthCheck` | Keep-alive / системный статус устройства | `310`, `313`, `314`, `323`, `336`, `339`, `340` | `Platerra`, `Siplite` |
| `46` | `UartToCloud` | Данные с UART-порта (шлюз UART→MQTT) | `334`, `335` | `Siplite` |
| `53` | `Stm32FwUpdateResult` | Результат прошивки сопроцессора STM32 | `347`–`352` | `Siplite` |
| `54` | `SipRegStateChange` | Изменение состояния SIP-регистрации | `353`, `354`, `340` | `Siplite` |
| `63` | `FullStateNotify` | 🆕 Полный слепок состояния всех ячеек | `360` | `Platerra`, `Siplite` |
| `65` | `SeedInitDone` | 🆕 Завершение seed-инициализации NVS | `370` | `Siplite` |
| `70` | `L4HmiEvent` | 🆕 Результат задания по обновлению UI-каталога `l4-hmi` | `401`–`410` | `l4-hmi` |

> 📖 Полный справочник типов с подробными примерами payload — в [`event-types-reference.md`](./event-types-reference.md).
> Описание всех числовых тегов (`3xx`, `4xx`) — в [`event-property-tags.md`](./event-property-tags.md).
> Клиентам REST API рекомендуется **игнорировать неизвестные `event_type_code`** и ориентироваться на матрицу совместимости при маршрутизации событий разных классов устройств.

## 🔁 Идемпотентность

Сервер обеспечивает идемпотентную обработку входящих событий. Дедупликация выполняется по составному ключу:

```
(device_id, dev_event_id, dev_timestamp)  при условии  dev_event_id != 0
```

| Ситуация | Сохранение в БД | Вебхук | EVA-подтверждение устройству |
|----------|:---:|:---:|:---:|
| ✅ Новое событие | ✅ Да | ✅ Да | ✅ Да |
| 🔁 Дубликат (повторная доставка) | ❌ Нет | ❌ Нет | ✅ Да |

> 📖 Подробнее о механизме EVA и идемпотентности на уровне MQTT — в [`event-protocol-mqtt.md`](./event-protocol-mqtt.md#подтверждение-получения-eva).

### Форматы `dev_timestamp` (из MQTT User Properties)

Поле `dev_timestamp` из User Properties MQTT-сообщения принимается в форматах:

| Формат | Пример |
|--------|--------|
| Unix epoch (число или строка) | `1740984093` |
| ISO 8601 с TZ | `2026-02-18T01:43:16+03:00` |
| Отсутствует / нераспознан | используется текущее серверное время |

В API поле `timestamp` всегда возвращается в **ISO 8601 с TZ-смещением**.

## 💡 Рекомендации по интеграции

### Выбор эндпоинта под задачу

| Задача | Эндпоинт | Ключевые параметры |
|--------|----------|--------------------|
| Мониторинг всех событий устройства | `incremental` | `device_id`, `last_event_id` |
| Мониторинг событий всех устройств организации | `incremental` | только `last_event_id` |
| Фильтрация по типу события | `device-events/` | `events_include=[13,14]` |
| Временной ряд значений конкретного тега | `fields/` | `event_type_code`, `tag`, `interval_m` |
| Push-уведомления (без polling) | [Вебхуки](./3-webhooks.md) | — |

### Общие рекомендации

- 🔄 **Инкрементальное чтение:** Используйте `incremental` + сохраняйте `last_event_id` локально — это обеспечит восстановление после сбоя без пропуска событий.
- 🏷️ **Фильтрация:** Применяйте `events_include` / `events_exclude` чтобы снизить нагрузку и объём обрабатываемых данных.
- 📊 **Временные ряды:** Эндпоинт `fields/` оптимален для аналитики — не нужно скачивать полные объекты событий.
- 📦 **Декодирование payload:** Клиент самостоятельно интерпретирует теги в `payload["300"]`. Используйте [`event-property-tags.md`](./event-property-tags.md) как справочник.
- 🕐 **Временны́е метки:** Поле `timestamp` в ответах API всегда нормализовано к ISO 8601 с TZ, даже если в `payload["102"]` формат отличается (например, `ZMSK`).
- 🔔 **Push вместо polling:** Для получения событий без задержки — используйте [вебхуки](./3-webhooks.md).
