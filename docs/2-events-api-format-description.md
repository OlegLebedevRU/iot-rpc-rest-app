# Протокол API для получения событий устройств (evt)

> **Файл:** `docs/event-protocol-api.md`  
> **Версия:** 0.50  
> **Дата:** 2026  
> **Тема:** Формат асинхронных событий, доступных через REST API

---

## Назначение

Данный документ описывает **формат данных и интерфейс доступа к событиям устройств** через REST API. В отличие от протокола MQTT, события здесь уже получены сервером из топика `dev/<SN>/evt`, сохранены в базе данных и доступны клиенту (пользователю API) в исходном виде.

API не переименовывает поля и не преобразует структуру `payload`. Вместо этого — сервер сохраняет **оригинальное JSON-сообщение от устройства как есть**, а клиент может фильтровать и запрашивать события по их числовым тегам.

### Идемпотентность

Сервер обеспечивает идемпотентную обработку входящих событий. Дедупликация выполняется по составному уникальному ключу `(device_id, dev_event_id, dev_timestamp)` при условии `dev_event_id != 0`. Повторные события с одинаковым ключом не создают дублирующих записей в базе данных.

### Поддерживаемые форматы `dev_timestamp` (на уровне MQTT User Properties)

Поле `dev_timestamp` из User Properties MQTT-сообщения может быть передано в одном из форматов:
- **Unix epoch** (целое число или строка): `1740984093`
- **ISO 8601** (строка с часовым поясом): `2026-02-18T01:43:16+03:00`
- Если значение отсутствует или не может быть разобрано — используется текущее серверное время.

В API (`timestamp`) поле отдаётся в формате ISO 8601 с TZ-смещением.

### Подтверждение получения (EVA)

После обработки события сервер отправляет подтверждение `EVA` в топик `srv/<SN>/eva` (см. [`event-protocol-mqtt.md`](./event-protocol-mqtt.md)). EVA отправляется как для новых событий, так и для идемпотентных дубликатов.

## Общая структура события (DevEventOut)

Событие — это JSON-объект, содержащий как метаданные сервера, так и **оригинальное сообщение от устройства** в поле `payload`.

### Поля события:

| Поле | Тип | Описание |
|------|-----|--------|
| `id` | `int` | Уникальный ID события в системе (автоинкремент) |
| `device_id` | `int` | Внутренний ID устройства в системе |
| `sn` | `string` | Серийный номер устройства (`<SN>` из топика) |
| `event_type_code` | `int` | Код типа события (значение тега `200`) |
| `timestamp` | `datetime` | Время создания события на устройстве (из `102`) |
| `received_at` | `datetime` | Время получения события сервером |
| `dev_event_id` | `int` | Идентификатор события на устройстве (тег `101`) |
| `correlation_id` | `string` | UUID события (из User Properties MQTT) |
| `payload` | `object` | Оригинальное JSON-сообщение от устройства |

> Поле `payload` содержит **полный оригинальный объект**, как он был отправлен устройством в MQTT.

---

## Пример события в API

````json 
{ "id": 150283, "device_id": 4712, "sn": "a3b1234567c10221d290825", "event_type_code": 13, "timestamp": "2026-02-18T01:43:16+03:00", "received_at": "2026-02-18T01:43:17Z", "dev_event_id": 20338, "correlation_id": "d9afcbfa-3d2c-4304-8e69-f644fef29f1f", "payload": { "101": 20338, "102": "2026-02-18T01:43:16ZMSK", "200": 13, "300": [ { "304": 12, "305": 1, "306": 22, "307": 35921924, "308": 8392849 } ] } }
````
> Обратите внимание: `payload` — это **не декодированный объект**, а точная копия MQTT-сообщения.

## Доступные API-эндпоинты

### 1. `GET /api/v1/device-events/` — Поиск событий по устройству

Получение списка событий с пагинацией.

#### Параметры запроса:
| Параметр | Тип | Обязательный | Описание |
|---------|-----|-------------|----------|
| `device_id` | `int` | Да | ID устройства |
| `events_include` | `list[int]` | Нет | Фильтр: только указанные коды событий (`200`) |
| `events_exclude` | `list[int]` | Нет | Исключить указанные коды событий |

#### Ответ:
````json 
{ "items": [ { /* DevEventOut */ } ], "total": 150, "page": 1, "size": 50, "pages": 3 }
````
---

### 2. `GET /api/v1/device-events/incremental` — Инкрементальные события

Получение новых событий с момента последнего запроса.

#### Параметры запроса:
| Параметр | Тип | Обязательный | Описание |
|---------|-----|-------------|----------|
| `device_id` | `int` | Нет | Если не указан — события по всем устройствам организации |
| `last_event_id` | `int` | Нет | Последний известный ID события. Если не указан — используется сохранённый оффсет |
| `limit` | `int` (1–100) | Нет | Максимальное количество событий (по умолчанию 50) |

#### Ответ:
````json
 [ { "id": 150283, "device_id": 4712, "sn": "a3b1234567c10221d290825", "event_type_code": 13, "timestamp": "2026-02-18T01:43:16+03:00", "received_at": "2026-02-18T01:43:17Z", "dev_event_id": 20338, "correlation_id": "d9afcbfa-3d2c-4304-8e69-f644fef29f1f", "payload": { "101": 20338, "102": "2026-02-18T01:43:16ZMSK", "200": 13, "300": [ { "304": 12 } ] } } ]
````
> Подходит для периодического опроса (polling). События приходят в порядке возрастания `id`.

---

### 3. `GET /api/v1/device-events/fields/` — Выборка значений тегов

Позволяет извлекать значения конкретных **числовых тегов** из `payload` за заданный интервал времени.
Подходит не только для телеметрии, но и для получения события в режиме polling, когда нужно регулярно проверять конкретный `event_type_code` и значение тега.

#### Параметры запроса:
| Параметр | Тип | Обязательный | Описание |
|---------|-----|-------------|----------|
| `device_id` | `int` | Да | ID устройства |
| `event_type_code` | `int` | Да | Код типа события (`200`) |
| `tag` | `str` | Да | Числовой тег для извлечения (например, `"304"`) |
| `interval_m` | `int` | Да | Интервал времени в минутах (назад от текущего момента) |
| `limit` | `int` | Нет | Лимит записей (макс. 1000) |

#### Пример polling-запроса:
```bash
curl -X 'GET' \
  'https://dev.leo4.ru/api/v1/device-events/fields/?device_id=4617&event_type_code=13&tag=304&interval_m=5&limit=10' \
  -H 'accept: application/json' \
  -H 'x-api-key: ApiKey <key>'
```

#### Ответ:
````json 
[ { "created_at": "2026-02-18T01:43:16+03:00", "value": 12, "interval_sec": 8 }, { "created_at": "2026-02-18T02:15:08+03:00", "value": 5, "interval_sec": 41 } ]
````
> Для polling `CellOpenEvent` запрашивайте `event_type_code=13` и `tag=304`: если в ответе появилось нужное значение ячейки, событие получено.
> Значение берётся из первого объекта в массиве `300[tag]`, если присутствует.

## Коды типов событий (event_type_code)

Аналогично MQTT-протоколу. Ниже — основные коды.

| Код | Описание | Пример `payload` |
|-----|--------|------------------|
| `0` | `Hello` | `{ "101": ..., "200": 0, "300": [{ "310": "1.04.025", "311": 13 }] }` |
| `3` | `IdInput` | `{ "101": ..., "200": 3, "300": [{ "301": "044AFE42C76781", "302": 6 }] }` |
| `13` | `CellOpenEvent` | `{ "101": ..., "200": 13, "300": [{ "304": 12 }] }` |
| `14` | `CellCloseEvent` | `{ "101": ..., "200": 14, "300": [{ "304": 12 }] }` |
| `44` | `DevHelthCheck` | `{ "101": ..., "200": 44, "300": [{ "310": "1.04.025", "313": 65740 }] }` |
| `46` | `UartToCloud` | `{ "101": ..., "200": 46, "300": [{ "334": 0, "335": "SGVsbG8h" }] }` |
| `63` | `CellStatesSnapshot` | `{ "101": ..., "200": 63, "300": [{ "360": [{ "304": 1, "321": 1 }, ...] }] }` |

Значения тегов внутри `payload.300[*]` интерпретируются по [справочнику тегов свойств событий](./event-property-tags.md).

---

### Пример события типа `63` (состояние всех ячеек постамата)

````json
{
  "101": 502,
  "102": "2026-04-22T13:48:02+03:00",
  "200": 63,
  "300": [
    {
      "360": [
        { "304": 1, "321": 1 },
        { "304": 2, "321": 1 },
        { "304": 3, "321": 0 }
      ]
    }
  ]
}
````

> Тег `360` содержит список состояний ячеек: ячейки №1 и №2 открыты (`321: 1`), ячейка №3 закрыта (`321: 0`). Структура объектов внутри `360` определена в [справочнике тегов](./event-property-tags.md#теги-состояния-ячеек-постамата-360369).

## Рекомендации по использованию API

- Все события хранятся в **оригинальном формате** — клиент должен интерпретировать теги самостоятельно.
- Используйте `incremental` для получения новых событий без потерь.
- Используйте `fields` для сбора временных рядов (например, `304` — номер ячейки).
- Храните `last_event_id` локально для восстановления после обрыва соединения.
- Все временные метки в `timestamp` приведены к ISO 8601 с TZ-смещением.
- Для фильтрации используйте `events_include` и `events_exclude`.

---

## Заключение

API предоставляет прямой доступ к сырым MQTT-событиям:
- **Нет переименования полей**
- **Нет маппинга тегов**
- **Полный `payload` как в MQTT**
- **Поддержка инкрементального чтения и выборки по тегам**
- **Идемпотентная обработка** по ключу `(device_id, dev_event_id, dev_timestamp)`

Это делает API максимально гибким для различных клиентов, которые могут сами решать, как интерпретировать данные. Подходит для:
- Фронтенд-приложений с кастомной логикой
- Систем аналитики
- Интеграций с внешними сервисами
- Отладки устройств
